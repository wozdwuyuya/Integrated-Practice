# 项目规范与执行协议 (Agent Execution Protocol)

## 1. 核心工作流
- Explore First: 在未读取相关文件源码前，严禁给出具体代码建议。先使用 `ls` 或读取文件。
- Atomic Changes: 每次修改必须是原子的。优先修改逻辑，再修改配置，最后更新文档。
- Pre-computation: 在执行修改前，模拟该修改是否会导致依赖项报错。
- Tool-Loop: 执行代码后必须查看 exit code 和 logs。如果报错，自动触发反思阶段，寻找 Root Cause 并在修改前向我确认。

## 2. 思考与输出格式
遇到复杂架构或逻辑错误时，必须进行"内部竞标"模拟 A/B 方案，并在回答前输出 `[Thinking Process]`，验证逻辑链路消除幻觉后，再输出最终代码。

## 3. 验证与测试闭环
完成代码编写后，不要干等。主动搜索项目中对应的测试脚本或编译命令，尝试在沙箱内运行它，通过真实的返回结果来验证你的代码是否正确。

## 4. SDK 参考强制路径
当需要开发硬件驱动、SLE(星闪)通信、或北向通信功能时，严禁凭空捏造 API。必须优先使用 `cat` 或 `ls` 查阅以下目录中的官方范例代码与头文件：
- `references/bearpi_3863_sdk/` — BearPi-Pico H3863 官方 SDK（驱动、协议栈、示例）
- `references/northbound_docs_v5_7/` — 北向通信协议文档与传感器数据分析范例

## 5. 编译对齐要求
确保新写的代码风格、宏定义、CMake 引用关系与 `references/bearpi_3863_sdk/` 中的原厂范例保持一致。禁止引入原厂 SDK 中不存在的自定义宏或命名约定。
